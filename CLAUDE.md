## Architecture

**API-only** — no frontend templates. Django admin UI is the only HTML interface. All application logic is exposed through REST endpoints under `/api/`.

**Settings hierarchy** — `config/settings/base.py` defines shared config; `local.py`, `test.py`, and `production.py` extend it. Active layer is controlled by `DJANGO_SETTINGS_MODULE`. Tests always use `config.settings.test`.

**Databases**

- Main DB: `cantoral` — default `DATABASE_URL=postgres:///cantoral`
- Test DB: `cantoral_tests` — set via `DATABASES["default"]["TEST"]["NAME"]`; pytest-django creates/tears it down automatically.

**JWT Authentication** — `djangorestframework-simplejwt` with refresh-token rotation and blacklisting.

- `POST /api/auth/token/` — obtain access + refresh tokens (body: `{"email": "...", "password": "..."}`)
- `POST /api/auth/token/refresh/` — rotate refresh token
- `POST /api/auth/token/verify/` — verify token validity
- All protected endpoints require `Authorization: Bearer <access_token>`

**Custom User model** (`cc/users/models.py`)

- Extends `AbstractUser`; no `username` field
- `USERNAME_FIELD = "email"` — simplejwt uses email for login automatically
- `UserManager` handles `create_user` / `create_superuser` with email

**DRF setup** — `config/api_router.py` registers viewsets with `DefaultRouter` (DEBUG) or `SimpleRouter` (production). Add new apps by registering their router there. OpenAPI schema auto-generated by drf-spectacular at `/api/schema/` (admin-only).

**App layout** — each app lives under `cc/<appname>/`:

```
cc/<appname>/
  models.py
  admin.py
  services.py -> read more about it in (./docs/app/services.md)
  api/
    serializers.py
    views.py
  tests/
    factories.py
    test_models.py
    api/
      test_views.py
```

Register new apps in `LOCAL_APPS` in `config/settings/base.py` and in `config/api_router.py`.

## Service / Serializer pattern

Every non-trivial endpoint follows this three-layer flow:

```
View → Serializer (validate input only) → Service (business logic, atomic)
```

**Serializers** — validate only. No DB writes, no side effects, no business logic.
Field validators (`validate_<field>`) may query the DB to check existence but must not modify it.

**Services** — one class per operation, always in `cc/<app>/services.py`:

```python
class XService:
    def __init__(self, param1: str, param2: str) -> None:
        self.param1 = param1
        self.param2 = param2

    @transaction.atomic
    def dispatch(self) -> ...:
        ...
```

Rules:
- Always decorated with `@transaction.atomic` so a failure rolls back cleanly.
- Send emails (or any external I/O) via `transaction.on_commit(...)` so they only fire if the transaction commits.
- Raise plain Python exceptions (`ValueError`) for domain errors; the view maps them to `ApiResponse`.

Use a service when the operation is complex (multiple DB writes) or reusable across views.
For simple reads or single-step writes, the serializer or view is enough.

**Views** — thin. Validate input, call the service, map exceptions to `ApiResponse`:

```python
def post(self, request: Request) -> ApiResponse:
    serializer = XSerializer(data=request.data)
    if not serializer.is_valid():
        return ApiResponse(errors=serializer.errors, success=False, status=400)
    try:
        result = XService(**serializer.validated_data).dispatch()
    except ValueError as exc:
        return ApiResponse(errors=[str(exc)], success=False, status=400)
    return ApiResponse(data=result, status=200)
```

## Email classes

Never call `send_mail()` directly. Every transactional email is a class in `cc/<app>/emails.py` that extends `BaseEmail`:

```python
from cc.utils.emails import BaseEmail

class WelcomeMail(BaseEmail):
    subject = "Welcome!"
    template_name = "users/emails/welcome"  # no extension

    def __init__(self, to: str, name: str) -> None:
        super().__init__(to=to, name=name)
```

Two template files are required per email: `<template_name>.html` and `<template_name>.txt`, placed under `cc/<app>/templates/`.

Call `.send()` only from inside `transaction.on_commit(lambda: ...)` in a service so the email fires only after the DB transaction commits.

## Celery

All external I/O (email, webhooks, slow operations) must be a Celery task — never inline in a view or service.

- Tasks live in `cc/<app>/tasks.py` and use `@shared_task`.
- Always trigger via `transaction.on_commit(lambda: my_task.delay(...))` inside a `@transaction.atomic` service.
- The generic `send_email_task` in `cc/utils/tasks.py` handles all email sending — use `BaseEmail` subclasses instead of calling it directly.
- In tests, `CELERY_TASK_ALWAYS_EAGER = True` makes tasks run synchronously.

## Rate limiting

Auth endpoints (`RegisterView`, `LoginView`, `VerifyEmailTokenView`) must declare `throttle_classes = [AuthRateThrottle]`. The `auth` scope allows 10 requests/minute. General API throttling (100/hour anon, 1000/hour user) is applied globally via DRF settings.

## Content-Type

`JSONContentTypeMiddleware` enforces `Content-Type: application/json` on all `POST`/`PUT`/`PATCH` requests to `/api/`. Returns HTTP 415. No per-view checks needed.

## Code conventions

- **Typed Python** everywhere: all models, serializers, views, and utilities must be fully annotated. `mypy` enforces this with `django-stubs` + `drf-stubs`.
- **Ruff** is the sole linter and formatter — broad rule set in `pyproject.toml`. `ruff check --fix` resolves most issues automatically.
- **Migrations** are excluded from mypy and ruff.
- **Tests** use pytest fixtures, not `unittest.TestCase`. Factories live in `cc/<app>/tests/factories.py` (factory-boy).
- **No HTML templates** — do not add template-based views, forms, or crispy-forms. Any new endpoint must be a DRF APIView or ViewSet.
- `ATOMIC_REQUESTS = True` wraps every request in a DB transaction — avoid long operations in views.
- We use DRY Principle.
